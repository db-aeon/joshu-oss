# Hermes Customizations

This file is the canonical Joshu repo documentation for how Hermes is integrated
and customized. Repo docs belong under `docs/` only — not README files under
`integrations/` or other runtime paths.

This repo intentionally treats Hermes as an upstream dependency, not as code we
normally vendor or fork.

Hermes changes quickly, and Joshu should benefit from that velocity. Joshu-owned
behavior should live in Hermes-supported customization points whenever possible:
skills, plugins, scripts, prompts, environment variables, and config generated by
Joshu startup code.

## Where Hermes Lives

Local development expects a separate Hermes checkout:

```text
/Users/danbenyamin/Documents/dev/hermes-agent
```

The local binary path is configured with `HERMES_BIN`, usually:

```text
/Users/danbenyamin/Documents/dev/hermes-agent/venv/bin/hermes
```

The sandbox image fetches a pinned Hermes checkout during image build from upstream:

```text
https://github.com/NousResearch/hermes-agent.git
```

The pinned ref is configured in `deploy/RELEASE.json` as `HERMES_AGENT_REF` and can be
overridden at deploy time. The pinned checkout must include Hermes's generic
Camofox `adopt_existing_tab` support.

In the sandbox image, Hermes is installed at:

```text
/opt/hermes-agent
```

and `deploy/scripts/vps-start.sh` points `HERMES_BIN` at:

```text
/opt/hermes-agent/venv/bin/hermes
```

Hermes runtime state lives under `$HERMES_HOME`:

- Local: usually `~/.hermes`
- Sandbox image: `/root/.hermes`, backed by the `joshu-hitl-hermes-home` Docker volume

### Pip extras parity (voice / gateway)

[`deploy/RELEASE.json`](../deploy/RELEASE.json) and [`scripts/update-hermes-agent.sh`](../scripts/update-hermes-agent.sh) share the same **`HERMES_IMAGE_EXTRAS`** list so the local Hermes venv and the VPS sandbox image install matching optional stacks. That list includes **`voice`** (CLI STT/TTS) and **`messaging`** (Discord/Telegram gateway, including Discord voice dependencies). The same **`[voice]`** stack backs **Hermes Chat** browser mic + TTS via Joshu’s **`/api/hermes-chat/transcribe`** and **`/tts`** routes (see [`docs/hermes-chat-arozos-app.md`](hermes-chat-arozos-app.md)).

System libraries for FFmpeg, Opus, and PortAudio are installed in the VPS sandbox image; locally on macOS use Homebrew per `docs/local-installation.md` (**Voice** subsection). Voice-mode env keys (for example **`GROQ_API_KEY`**) and gateway tokens belong in `~/.hermes/.env` locally and in the sandbox image via **`HERMES_ENV_B64`**, not in git.

### Telegram 1:1 chat (Hermes messaging gateway)

Joshu runs **`hermes gateway run`** (see below). When Hermes messaging env vars are set, the gateway attaches a **second** Telegram bot for owner ↔ Joshu chat. This is separate from the **action-guard** bot (`JOSHU_ACTION_GUARD_TELEGRAM_BOT_TOKEN` in [`connectors.md`](connectors.md#action-guard-owner-approval-for-writes)) — use two BotFather bots so approval polling and chat polling do not fight over one token.

| Bot | Env | Purpose |
| --- | --- | --- |
| **Joshu** (action guard) | `JOSHU_ACTION_GUARD_TELEGRAM_BOT_TOKEN` + `JOSHU_ACTION_GUARD_TELEGRAM_ALLOWED_USERS` | Approve / deny gated writes (email send, …) |
| **Hermes** (messaging) | `TELEGRAM_BOT_TOKEN` + `TELEGRAM_ALLOWED_USERS` in `~/.hermes/.env` | Full agent chat — tools, skills, voice memos, cron delivery |

**Setup (local or VPS):**

1. Create a **new** bot in [@BotFather](https://t.me/BotFather) (`/newbot`). Name it for chat (e.g. “Patrick’s Joshu”), not the approvals bot.
2. Get the owner’s numeric Telegram user ID ([@userinfobot](https://t.me/userinfobot)). Welcome `communicationContacts.telegram` is only a username hint — authorization uses the numeric ID.
3. Add to `~/.hermes/.env` (or repo `.env` / VPS `/etc/joshu/instance.env` — Joshu syncs these into `~/.hermes/.env` on gateway start):

```bash
TELEGRAM_BOT_TOKEN=123456789:ABCdefGHIjklMNOpqrSTUvwxYZ
TELEGRAM_ALLOWED_USERS=123456789
```

4. Restart the Hermes gateway (`hermes gateway stop`, then restart Joshu / `dev:arozos`, or on VPS: `curl -fsS 'http://127.0.0.1:8788/joshu/api/hermes-chat/status?after_mcp_boot=1'`). Gateway logs should show `✓ telegram connected`.
5. DM the **chat** bot on Telegram. Sessions are keyed `agent:main:telegram:dm:<chat_id>` (Hermes session store).
6. Optional: send `/sethome` in that DM so Hermes cron / scheduled task output delivers there.

**Wizard alternative:** `hermes gateway setup` → Telegram (writes the same env/config).

**VPS:** Always-on boxes use Hermes **long polling** by default (no extra Caddy route). Webhook mode (`TELEGRAM_WEBHOOK_URL` + `TELEGRAM_WEBHOOK_SECRET`) is only needed for sleep-when-idle cloud hosts — see [Hermes Telegram docs](https://hermes-agent.nousresearch.com/docs/user-guide/messaging/telegram).

**Verify:**

- Gateway log: `[Telegram] Connected to Telegram (polling mode)` in `~/.hermes/logs/gateway.log`
- Bot identity: `curl -fsS "https://api.telegram.org/bot<TOKEN>/getMe"`
- Langfuse traces for Telegram turns use source `telegram` and the same Hermes toolsets as jChat
- Action-guard status (other bot): `GET /joshu/api/action-guard/status` → `telegramLinked`, `telegramAllowlistConfigured`

#### jChat vs Telegram (same gateway, different pipes)

Both surfaces talk to the **same** `hermes gateway run` process on `:8642` — not separate agents.

| | jChat | Telegram chat bot |
| --- | --- | --- |
| **Ingress** | Joshu `POST /api/hermes-chat/stream` → `:8642/v1/chat/completions` (**api_server**) | Hermes **telegram** platform (long polling) |
| **Session key** | `joshu-hermes-chat:<sessionId>` | `agent:main:telegram:dm:<chat_id>` |
| **UI** | ArozOS jChat subservice | Telegram app |

Details: [hermes-chat-arozos-app.md — Request path](hermes-chat-arozos-app.md#request-path-jchat-vs-telegram-vs-slack).

### Slack chat (Hermes messaging gateway)

Hermes supports Slack via **Socket Mode** (no public webhook). This is separate from **Composio Slack** (MCP tools) and the **owner approval** channel (Composio Y/N).

| Integration | Purpose |
| --- | --- |
| **Hermes Slack** (`SLACK_BOT_TOKEN` + `SLACK_APP_TOKEN`) | Full agent chat — DM or @mention in channels |
| **Composio Slack** | Agent tools (`SLACK_SEND_MESSAGE`, …) |
| **Owner 1:1 Slack** | Write approvals only |

**Joshu setup (recommended):** **Safety** app → **Hermes Slack chat**:

1. **Generate manifest** — shows JSON in the UI (companion name from `.joshu/identity.json`); also writes `~/.hermes/slack-manifest.json`. Create the app at [api.slack.com](https://api.slack.com/apps). Slash-command `url` fields (`hermes-agent.local`) are schema placeholders — Socket Mode ignores them.
2. Enable **Socket Mode** → app-level token `xapp-…` with `connections:write`.
3. **App Home → Messages Tab** (for DMs).
4. **OAuth & Permissions** → Install → bot token `xoxb-…`.
5. Paste **xoxb** / **xapp** + your member ID (`U…` in `SLACK_ALLOWED_USERS` — required).
6. **Save** → **Restart gateway** (Safety button, or confirm on Save when messaging changed).

Joshu stores secrets in `.joshu/safety-settings/local-env.json` and syncs `SLACK_*` into `~/.hermes/.env` on save (same pattern as `TELEGRAM_BOT_TOKEN`).

**Multiple Joshu boxes in one Slack workspace:** one Slack app per box (unique `xoxb` / `xapp`). Do not reuse the same `xapp-` token on two machines — Socket Mode allows one connection per app. Use separate channels/DMs per companion.

```bash
SLACK_BOT_TOKEN=xoxb-…
SLACK_APP_TOKEN=xapp-…
SLACK_ALLOWED_USERS=U0123456789
SLACK_HOME_CHANNEL=C01234567890      # optional — cron / proactive delivery
SLACK_ALLOWED_CHANNELS=C01234567890  # optional — restrict channel @mentions
```

Hermes docs: [Slack messaging](https://hermes-agent.nousresearch.com/docs/user-guide/messaging/slack). CLI alternative: `hermes slack manifest --write` then `hermes gateway setup`.

#### Hindsight `<memory-context>` (not Telegram-specific)

When Hindsight is enabled (`memory.provider: hindsight`), Hermes **auto-recalls** relevant memories before each turn and injects them into the **current turn’s user message** wrapped in `<memory-context>` (Hermes `agent/memory_manager.py` — `build_memory_context_block()`). This applies to **jChat, Telegram, cron, and API** alike; it is not a Telegram-only prompt.

- Documented: [Nous Memory Providers](https://hermes-agent.nousresearch.com/docs/user-guide/features/memory-providers), [Hindsight ↔ Hermes](https://hindsight.vectorize.io/sdks/integrations/hermes)
- Tune on VPS via `HINDSIGHT_AUTO_RECALL`, `HINDSIGHT_MEMORY_MODE` (`hybrid` / `context` / `tools`), `HINDSIGHT_RECALL_BUDGET` in `instance.env` → `~/.hermes/hindsight/config.json` (Joshu writes this in `src/hermesApi.ts`)
- If the block appears in **visible** chat output, that is usually model echo; Hermes strips it from streamed deltas via `StreamingContextScrubber` (#5719)

## What Joshu Owns

Joshu-specific Hermes customization should be versioned in this repo:

```text
integrations/hermes/
├── skills/                  # factory skill source (seeded at boot — see below)
├── skills-enabled.yaml      # product allowlist (computes bundled denylist)
└── skills-disabled.yaml     # optional explicit bundled blocks (usually empty)
.hermes/plugins/              # repo-owned project plugins
patches/                      # temporary upstream patches
```

Bundled Hermes skills (`joshu-browser`, `kanban-worker`, `mcporter`, …) live in the
Hermes checkout — not under `integrations/hermes/skills/`.

Hermes skills and plugins should be preferred over core patches. Carry a patch
only when Hermes does not yet expose the extension point Joshu needs.

Joshu-owned Hermes core patches (under `scripts/`):

| Patch | Target | Apply script |
| --- | --- | --- |
| Camofox HITL (legacy) | `tools/browser_camofox.py` | `apply-hermes-hitl-patch.sh` |
| Langfuse tracing | `plugins/observability/langfuse/__init__.py` | `apply-hermes-langfuse-patches.sh` |
| Skill evolution ledger | `tools/skill_manager_tool.py` | `apply-hermes-skill-evolution-patch.sh` |
| Provider `content_filter` | `run_agent.py` | `apply-hermes-content-filter-patch.sh` |
| Langfuse agent-loop `post_tool_call` | `run_agent.py` | `apply-hermes-invoke-tool-post-hook-patch.sh` |

The legacy Hermes core patch is:

```text
scripts/hermes-browser-camofox-hitl.patch
```

It adapts older Hermes checkouts' Camofox browser tool to the shared HITL Camofox
identity. Current Hermes checkouts already have the generic configuration hook,
so this patch is kept only as a fallback for old clones.

The upstream path was tracked in:

```text
https://github.com/NousResearch/hermes-agent/pull/23475
```

The generic path uses Hermes/Camofox names instead of Joshu-specific
`HITL_CAMOFOX_*` names:

```text
CAMOFOX_USER_ID
CAMOFOX_SESSION_KEY
CAMOFOX_ADOPT_EXISTING_TAB
```

and matching config keys:

```yaml
browser:
  camofox:
    user_id: shared-camofox
    session_key: visible-tab
    adopt_existing_tab: true
```

The intention is to support HITL Camofox workflows where a human operator and
Hermes share the same visible browser session. Joshu sets these env vars when it
starts the Hermes gateway so Hermes adopts the visible tab instead of creating a
second hidden tab/session.

## HITL Patch Lifecycle

Newer Hermes checkouts include the generic Camofox identity/adoption controls, so
Joshu sets both the current generic env names and the older `HITL_CAMOFOX_*`
names for compatibility:

```text
CAMOFOX_USER_ID=hitl-camofox
CAMOFOX_SESSION_KEY=<the shared Joshu session key>
CAMOFOX_ADOPT_EXISTING_TAB=true
```

The legacy patch is kept only as a fallback for older Hermes checkouts. The local
helper skips cleanly when generic support is already present:

```bash
cd /Users/danbenyamin/Documents/dev/joshu
scripts/apply-hermes-hitl-patch.sh
```

Joshu runs this automatically on startup (`npm run dev`, `npm run dev:arozos`) before
the Hermes gateway starts. The script is idempotent: it no-ops when the checkout
already has tab adoption. After `scripts/update-hermes-agent.sh` refreshes Hermes,
the next Joshu start re-applies the patch if upstream removed it.

The image build verifies that the pinned Hermes checkout includes baseline Camofox tab support
(adoption **`adopt_existing_tab`** **or** the **`_ensure_tab` + `sessionKey`** REST shape)
and fails the build if neither is present. Joshu no longer applies the legacy Hermes
patch; only Camofox itself is patched in the VPS sandbox image.

## Provider `content_filter` handling

Some OpenRouter backends (notably **DeepSeek**) return `finish_reason: content_filter`
with provider moderation boilerplate instead of a real answer. A common DeepSeek pattern
is Chinese text like `你好，我无法给到相关内容。` (“Hello, I cannot provide relevant
content.”). Hermes upstream passes that through to Telegram/jChat today.

Joshu patches **`run_agent.py`** so filtered completions are retried instead of
delivered to the user:

1. **Detect** `finish_reason: content_filter` and known refusal patterns.
2. **Discard** streamed moderation text (do not append it to session history).
3. **Retry up to 2×** with bulky `role: tool` payloads truncated (~4K chars) and a
   synthetic user nudge to respond in English and summarize tool output.
4. **Try `fallback_providers`** if configured in `$HERMES_HOME/config.yaml`.
5. **Return a clear English error** if retries and fallback are exhausted.

Files:

```text
scripts/hermes-content-filter.patch
scripts/apply-hermes-content-filter-patch.sh
```

Applied automatically on `npm run dev:arozos`, `npm run hermes:update`, VPS boot
(`deploy/scripts/vps-start.sh`), sandbox image build (`deploy/Dockerfile`), and
gateway warm-up (`ensureJoshuHermesConfig()` in `src/hermesApi.ts` — restarts the
gateway when the patch is newly applied).

Manual apply (idempotent — skips when `_is_provider_content_filter_response` is
already present in `run_agent.py`):

```bash
cd /Users/danbenyamin/Documents/dev/joshu
HERMES_DIR=/Users/danbenyamin/Documents/dev/hermes-agent \
  scripts/apply-hermes-content-filter-patch.sh
hermes gateway stop   # gateway must reload run_agent.py
```

**Verify:** In Langfuse, a blocked attempt still shows `finish_reason: content_filter`
on the failed generation; a successful retry appears as a later LLM call in the same
Hermes turn.

**If DeepSeek keeps blocking after retries:** add a `fallback_providers` entry (e.g.
`google/gemini-flash-latest` on OpenRouter) — see [Hermes fallback providers](https://hermes-agent.nousresearch.com/docs/user-guide/features/fallback-providers).
Long-term fix is upstreaming content-filter retry logic to Hermes.

## Runtime Wiring

Joshu auto-starts the Hermes gateway when needed. The user normally does not run
the gateway separately.

`src/hermesApi.ts` starts:

```text
hermes gateway run --replace --quiet --accept-hooks
```

Before starting the gateway, Joshu runs `ensureJoshuHermesConfig()` so
`$HERMES_HOME/config.yaml` includes the Joshu skill dir, toolsets, Camofox
identity, and the model block from `JOSHU_HERMES_MODEL` / `JOSHU_HERMES_PROVIDER`
in the Joshu `.env` (or process env). Personal keys (`voice`, `stt`, `tts`, …)
live in `$HERMES_HOME/config.user.yaml`; Joshu merges them into `config.yaml`
only when the merged result differs from disk ([`src/hermesConfigSplit.ts`](../src/hermesConfigSplit.ts)).
Joshu also syncs `ANTHROPIC_API_KEY`, `OPENROUTER_API_KEY`, and gateway tokens
into `$HERMES_HOME/.env` when present.

**Desktop `HERMES.md`:** Joshu writes a managed project-context file to
`terminal.cwd/HERMES.md` (ArozOS Desktop) so Hermes loads owner/assistant identity
and operating rules via `build_context_files_prompt()`. Implementation:
[`src/hermesContextFile.ts`](../src/hermesContextFile.ts). `ensureJoshuHermesConfig`
checks content on each gateway warm-up but **logs and writes only when identity,
profile, or env produce different markdown** — not on every health probe.

**`$HERMES_HOME/SOUL.md`:** Companion personality from portal onboarding (`soul_md`).
Generated at `/setup` with sections `# Identity`, `# Style`, `# Avoid`, `# Defaults`
per the [Hermes SOUL.md guide](https://hermes-agent.nousresearch.com/docs/guides/use-soul-with-hermes)
(identity and voice only — not tasks, tools, or user quiz facts). Provision writes
`/etc/joshu/secrets/companion-soul.md`; boot sync copies to `SOUL.md` with marker
`<!-- joshu-managed: companion-soul -->`. Implementation:
[`src/hermesSoulFile.ts`](../src/hermesSoulFile.ts),
[`src/companionIdentitySync.ts`](../src/companionIdentitySync.ts),
[`joshu-control-plane/apps/control-plane/src/lib/companion-forge/prompts.ts`](../joshu-control-plane/apps/control-plane/src/lib/companion-forge/prompts.ts).
See [joshu-identity.md](joshu-identity.md) and [control-plane-portal — Companion forge](https://github.com/db-aeon/joshu-control-plane/blob/main/docs/control-plane-portal.md#companion-forge).

### OpenRouter

Setup (also covered in the [OpenRouter Hermes cookbook](https://openrouter.ai/docs/cookbook/coding-agents/hermes-integration)):

1. API key at [openrouter.ai/keys](https://openrouter.ai/keys).
2. Either `hermes config set OPENROUTER_API_KEY sk-or-...` (`~/.hermes/.env`) or
   `OPENROUTER_API_KEY` in the Joshu `.env` (synced on gateway start).
3. `JOSHU_HERMES_PROVIDER=openrouter` and an OpenRouter model id in
   `JOSHU_HERMES_MODEL`.

Model IDs use `<author>/<slug>`; `~` picks the latest in a family (e.g.
`~anthropic/claude-sonnet-latest`). Browse [openrouter.ai/models](https://openrouter.ai/models).
Hermes requires **≥64K context** for the main model.

Joshu warns at gateway start when `JOSHU_HERMES_PROVIDER=openrouter` but
`OPENROUTER_API_KEY` is missing (`src/hermesApi.ts`).

Optional `~/.hermes/config.yaml` tuning:

- `openrouter.min_coding_score` with `openrouter/pareto-code` for cost-aware coding routing
- `provider_routing` (`sort`, `only`, `ignore`, `order`); shortcuts `:nitro` / `:floor` on model names
- `fallback_providers` for backup models mid-session
- `auxiliary.*` — Joshu sets `session_search` to OpenRouter + `google/gemini-3.1-flash-lite` (`JOSHU_HERMES_SESSION_SEARCH_*`); local dev uses `~google/gemini-flash-latest` for vision, compression, and titles

Monitor usage at [openrouter.ai/activity](https://openrouter.ai/activity). Restart the gateway after provider or key changes (`hermes gateway stop`, or restart Joshu dev).

### joshu-desktop plugin

Repo plugin [`.hermes/plugins/joshu-desktop/`](../.hermes/plugins/joshu-desktop/) exposes Hermes tool **`desktop_open`** for opening ArozOS modules (`jWeb`, `jMail`, …) and files under `joshu's files`. Joshu auto-adds toolset `joshu-desktop` when the plugin is enabled ([`src/hermesApi.ts`](../src/hermesApi.ts)).

| Piece | Role |
| --- | --- |
| `desktop_open` handler | Validates `kind` + `target`; returns JSON `{ ok, action }` |
| `post_tool_call` hook | Enqueues action to `POST /joshu/api/desktop-actions/enqueue` with `session_id` from Hermes |
| jChat / voice client | Executes `desktop_action` via [`desktopActions.ts`](../apps/hermes-chat/src/desktopActions.ts) |

**Why a hook:** Hermes `registry.dispatch()` passes only `task_id` / `user_task` to tool handlers — not `session_id`. The hook receives `session_id` so Joshu can key the queue as `joshu-hermes-chat:<id>`.

**Typed fast path** (`open the mail app`) bypasses Hermes entirely in jChat — see [hermes-chat-arozos-app.md — Desktop presentation](hermes-chat-arozos-app.md#desktop-presentation-two-tiers).

**Gateway reload:** After changing the plugin, restart Hermes (`hermes gateway stop`, then Joshu / `dev:arozos`).

### Disabled skills (product denylist)

Joshu computes **`skills.disabled`** at gateway sync from:

1. [`integrations/hermes/skills-enabled.yaml`](../integrations/hermes/skills-enabled.yaml) — product allowlist (~15 skills)
2. All bundled skill names discovered under the Hermes checkout (`HERMES_DIR` / `HERMES_BIN` → `skills/` and `optional-skills/`)
3. Joshu repo skills under `integrations/hermes/skills/` (auto-enabled)

Everything discovered that is **not** on the effective allowlist is written to `$HERMES_HOME/config.yaml` `skills.disabled` (~160 bundled Hermes skills blocked). The denylist is **not** committed to git — only the allowlist is.

**Hermes checkout resolution** ([`src/hermesSkillsConfig.ts`](../src/hermesSkillsConfig.ts) `resolveHermesAgentRoot()`):

| Env | VPS (Docker) | Local dev |
| --- | --- | --- |
| `HERMES_DIR` | `/opt/hermes-agent` (Dockerfile `ENV`) | Optional; defaults from `HERMES_BIN` |
| `HERMES_BIN` | `/opt/hermes-agent/venv/bin/hermes` | Required in `.env` — Joshu walks up from `venv/bin/hermes` until it finds `skills/` |

If bundled discovery fails (empty denylist), Hermes loads **all** ~170 upstream skills. Regression: `npm run test:hermes-skills-policy` (expects ~150+ disabled).

**Hermes learning loop:** After each turn, Hermes can spawn a background review that
uses `skill_manage` to create or update skills under `$HERMES_HOME/skills/`. Joshu
does **not** scan that directory on sync — new or evolved skills stay enabled unless
you add them to the denylist manually.

Refresh the allowlist after bumping the Hermes pin (updates comment counts in YAML):

```bash
npm run hermes:sync-skills-policy
npm run test:hermes-skills-policy
```

Optional per-environment extras: `JOSHU_HERMES_SKILLS_DISABLED=skill-a,skill-b` in
Joshu `.env` or `/etc/joshu/instance.env`. Optional explicit blocks (rare): [`integrations/hermes/skills-disabled.yaml`](../integrations/hermes/skills-disabled.yaml).

**Do not use an allowlist at runtime.** Joshu previously disabled every discovered skill
not on `skills-enabled.yaml` (~15 product skills enabled, ~160 bundled disabled). That blocked Hermes's
learning loop (`skill_manage` writing to `$HERMES_HOME/skills/`). Runtime policy is
**denylist-only** since **0.1.14**.

**Gateway sync lifecycle:**

- `ensureJoshuHermesConfig()` in [`src/hermesApi.ts`](../src/hermesApi.ts) merges `skills.disabled` on every gateway warm-up.
- When the denylist **changes**, Joshu **restarts the Hermes gateway** so the new catalog loads (same as plugin changes).
- **VPS boot:** [`deploy/scripts/vps-start.sh`](../deploy/scripts/vps-start.sh) nudges `GET /joshu/api/hermes-chat/status?after_mcp_boot=1`, then `verify_hermes_skills_denylist` re-nudges if `skills.disabled` count is &lt; 50.
- **Hard factory reset:** [`resyncHermesAfterBoxHardReset()`](../src/hermesApi.ts) re-seeds `skills/joshu/`, re-merges denylist, restarts gateway — no container restart required. Start a **new jChat** session after reset.

**Bootstrap vs learning (2026-06):** On a **release** bump, the default seed mode is

1. For each factory `SKILL.md` that differs from the box copy, `scripts/merge-hermes-factory-skill.mjs` calls OpenRouter (`JOSHU_HERMES_SKILLS_MERGE_MODEL`, default `google/gemini-3.1-flash-lite`) to produce a **per-box** merged `SKILL.md` — factory improvements plus this box’s evolved procedures.

4. **`overwrite`** (`rsync --delete`) runs only on **hard factory reset** via `resyncHermesAfterBoxHardReset()` → `JOSHU_HERMES_SKILLS_SEED_MODE=overwrite`.

### Skill catalog, descriptions, and `skill_view`

Hermes builds a **skills index** into every cached system prompt (`<available_skills>` in `agent/prompt_builder.py`). Joshu does **not** assemble or truncate this block — it comes from the pinned Hermes Agent runtime.

| Layer | Source | Content |
| --- | --- | --- |
| **Catalog line** | `description:` in each `SKILL.md` frontmatter | One line per skill in `<available_skills>` — this is what the model uses to pick a skill |
| **Full procedure** | `skill_view(name)` tool call | Entire `SKILL.md` (+ optional linked files via `skill_view(name, file_path)`) |

**60-character truncation (Hermes upstream):** `extract_skill_description()` in Hermes `agent/skill_utils.py` caps catalog text at **60 characters** (57 + `...`). Long `description` fields are **not** shown in full in the index — only after `skill_view`. Factory Joshu skills must keep `description` ≤ 60 chars with the routing hint in that one line (Hermes CONTRIBUTING hardline). Put detail in the skill body, not frontmatter.

**When does Hermes call `skill_view`?** There is no keyword router. The system prompt says to scan `<available_skills>` and **MUST** `skill_view(name)` when a skill matches or is partially relevant. The LLM decides each turn. Exceptions that **preload** skills without the model choosing:

- **Skill-backed crons** — `skills: ["ea-playbook"]` on the job; Hermes loads those skills at run start.
- **CLI `/skill-name`** or `build_preloaded_skills_prompt()` — full skill injected at session start.

**jChat** sends only a thin client system prompt ([`apps/hermes-chat/src/main.tsx`](../apps/hermes-chat/src/main.tsx)) plus the **current** user message; session history and the Hermes layers (SOUL, HERMES.md, skills index) live server-side via `X-Hermes-Session-Id`. Conversation drift (e.g. morning brief → drip campaign in one session) does not auto-switch skills — the model must re-scan the catalog and call `skill_view` for the new task type.

### Langfuse observability

Hermes ships a bundled [Langfuse](https://langfuse.com) plugin at
`observability/langfuse`. It traces turns, LLM calls, and tool usage (fail-open
when the SDK or credentials are missing).

**Local setup:**

```bash
# In the Hermes venv (same checkout as HERMES_BIN)
venv/bin/pip install langfuse
hermes plugins enable observability/langfuse
```

Add credentials to `~/.hermes/.env` (or set the same `HERMES_LANGFUSE_*` keys in
Joshu `.env` — Joshu syncs them into `$HERMES_HOME/.env` on gateway start):

```bash
HERMES_LANGFUSE_PUBLIC_KEY=pk-lf-...
HERMES_LANGFUSE_SECRET_KEY=sk-lf-...
HERMES_LANGFUSE_BASE_URL=https://us.cloud.langfuse.com
HERMES_LANGFUSE_ENV=local
HERMES_LANGFUSE_USER_ID=my-sandbox-slug
```

`HERMES_LANGFUSE_USER_ID` maps to Langfuse [User Tracking](https://langfuse.com/docs/observability/features/users) (`userId` on traces). On VPS sandboxes the control plane sets it to the **box slug** (hostname label before `CUSTOMER_DOMAIN_SUFFIX`, e.g. `voice-0531` for `voice-0531.box.joshu.me`).

Use the host that matches your project region (Settings in the Langfuse UI):
**US** → `https://us.cloud.langfuse.com`, **EU** → `https://cloud.langfuse.com`.
The `/api/public/health` check can succeed on both hosts even when trace
ingestion fails with 401 — always match `HERMES_LANGFUSE_BASE_URL` to your keys.

Joshu keeps the plugin enabled when `JOSHU_HERMES_PLUGIN_NAMES` includes
`observability/langfuse` (default in `.env.example`). Restart the Hermes gateway
after changing keys.

**Verify:** `hermes chat -q "hello"` then check the Langfuse UI for a "Hermes
turn" trace. See [Built-in Plugins — observability/langfuse](https://hermes-agent.nousresearch.com/docs/user-guide/features/built-in-plugins#observabilitylangfuse).

**VPS / Sandbox image:** `langfuse` is installed in the Hermes venv at image build;
Joshu applies `scripts/apply-hermes-langfuse-patches.sh` at image build and VPS boot
(system-prompt tracing + per-box `user_id`).

**Control plane (DO sandboxes):** set in `joshu-control-plane/apps/control-plane/.env.local` (or Vercel env):

```dotenv
DEFAULT_JOSHU_HERMES_PLUGIN_NAMES=observability/langfuse
DEFAULT_HERMES_LANGFUSE_PUBLIC_KEY=pk-lf-...
DEFAULT_HERMES_LANGFUSE_SECRET_KEY=sk-lf-...
DEFAULT_HERMES_LANGFUSE_BASE_URL=https://us.cloud.langfuse.com
DEFAULT_HERMES_LANGFUSE_ENV=production
```

New provisions copy these into `/etc/joshu/instance.env` plus `HERMES_LANGFUSE_USER_ID=<box-slug>` (e.g. `voice-0531`). **Boxes provisioned before Langfuse defaults existed** only have `HERMES_LANGFUSE_USER_ID` unless you edit `instance.env` manually — keys and `JOSHU_HERMES_PLUGIN_NAMES` are required for any traces.

**Gateway must reload plugins:** Hermes reads `plugins.enabled` only at gateway start. `vps-start.sh` runs `hermes plugins enable` when `JOSHU_HERMES_PLUGIN_NAMES` is set; Joshu restarts the gateway when it newly adds a plugin to `config.yaml`. If traces are empty despite keys in `instance.env`, run:

```bash
hermes plugins enable observability/langfuse
hermes gateway stop
# Joshu recreates gateway on next health/chat
```

**What shows up in Langfuse:** Hermes turns (`hermes chat`, jChat, voice `think` → Hermes SSE) — not health checks, Camofox alone, or OpenAI Realtime audio. Traces can appear without a **Users** row: `userId` requires `HERMES_LANGFUSE_USER_ID`, `scripts/hermes-langfuse-user-id.patch` (`trace_context` + `propagate_attributes`), gateway env (not only `~/.hermes/.env`), and a gateway restart after patch/keys change. Joshu sets `HERMES_LANGFUSE_USER_ID` on the gateway child and derives it from `CUSTOMER_DOMAIN` when unset. See [Users vs jChat traces](vps-sandbox/troubleshooting-and-lessons.md#langfuse-users-vs-jchat-traces).

**Joshu deterministic app (Node):** OpenRouter calls from the Joshu API (not Hermes) — Day 0 infer/sweep and the EA scheduling classifier — export via [`src/observability/langfuse.ts`](../src/observability/langfuse.ts) and [`src/day0/llm.ts`](../src/day0/llm.ts). Reuses the same `HERMES_LANGFUSE_*` keys (or standard `LANGFUSE_*` fallbacks). Fail-open when keys are absent.

| Trace name | When |
| --- | --- |
| `ea-scheduling-classifier` | Connector ingest classifies scheduling mail |
| `joshu-day0-infer` | Day 0 chunk summary + onboarding draft |
| `joshu-day0-sweep` | Day 0 incremental sweep digest |

All carry tag **`joshu-app`**. Local: restart Joshu after setting keys; look for `[joshu-langfuse] tracing enabled` in server logs. VPS: requires a **new sandbox image** when you add Node runtime deps (`@langfuse/tracing`, `@langfuse/otel`, `@opentelemetry/sdk-node` in [`deploy/runtime/package.json`](../deploy/runtime/package.json)) — dist-only hotpatch is not enough. See [hotpatch-running-box — Lane C](vps-sandbox/hotpatch-running-box.md#lane-c--full-image-release-only).

**System prompt in traces:** Anthropic passes `system` outside `messages`, so
Langfuse generation input omitted the Hermes system prompt until the hook passes
`system_prompt` through `pre_api_request`. Proposed upstream fix:
[NousResearch/hermes-agent#32175](https://github.com/NousResearch/hermes-agent/pull/32175)
(touches `agent/conversation_loop.py` on current `main`; older pins used

Until that ref is in your Hermes checkout, Joshu ships

the script if traces lose `role: system`. **Restart the gateway** after patch or
env changes so the plugin and OTLP exporter reload.

`role: system` (truncated to `HERMES_LANGFUSE_MAX_CHARS`, default 12000). Metadata

## Hindsight Memory

Joshu can use Hermes's native Hindsight memory provider for persistent
cross-session memory. Joshu runs Hindsight as an explicit local API process and
configures Hermes to use `local_external` mode, rather than letting Hermes start
an embedded daemon on first use. This keeps startup, health checks, logs, and
VPS persistence under Joshu's runtime scripts.

When enabled, `src/hermesApi.ts` adds:

```yaml
memory:
  provider: hindsight
```

and writes `$HERMES_HOME/hindsight/config.json` with a local API URL, usually:

```json
{
  "mode": "local_external",
  "api_url": "http://127.0.0.1:8888",
  "bank_id": "joshu",
  "memory_mode": "hybrid",
  "auto_recall": true,
  "auto_retain": true
}
```

Hindsight's service and database are local. The database is PostgreSQL. Local
development can use Hindsight's embedded pg0 path, while The sandbox image runs a local
PostgreSQL service in the same container and compiles pgvector into the image.
The LLM used by Hindsight for memory extraction and synthesis can be hosted or
OpenAI-compatible via `HINDSIGHT_API_LLM_*` environment variables.

The sandbox image installs `hindsight-api-slim[embedded-db]`, not the full local-ML package.
That keeps Hindsight lean and avoids bundling local embedding and reranker
models. CPU VPS sandboxes therefore requires external embedding and reranker configuration
before Hindsight memory starts. Local development can also use external
embeddings/reranking to keep behavior close to fleet sandboxes.

Startup behavior is controlled by:

```text
JOSHU_HINDSIGHT_ENABLED=auto|true|false
HINDSIGHT_API_URL=http://127.0.0.1:8888
HINDSIGHT_API_LLM_PROVIDER=openrouter
HINDSIGHT_API_LLM_API_KEY=<secret>   # falls back to OPENROUTER_API_KEY when unset
HINDSIGHT_API_LLM_MODEL=google/gemini-3.1-flash-lite
HINDSIGHT_API_LLM_BASE_URL=<optional OpenAI-compatible URL>
HINDSIGHT_API_EMBEDDINGS_PROVIDER=openai
HINDSIGHT_API_EMBEDDINGS_OPENAI_MODEL=text-embedding-3-small
HINDSIGHT_API_EMBEDDINGS_OPENAI_API_KEY=<secret>
# Or Gemini embeddings:
# HINDSIGHT_API_EMBEDDINGS_PROVIDER=google
# HINDSIGHT_API_EMBEDDINGS_GEMINI_API_KEY=<secret>
HINDSIGHT_API_RERANKER_PROVIDER=cohere
HINDSIGHT_API_RERANKER_COHERE_MODEL=rerank-english-v3.0
HINDSIGHT_API_RERANKER_COHERE_API_KEY=<secret>
```

Google Discovery Engine reranking also works, but it needs the Discovery Engine
API enabled and IAM access for the local ADC user or service account:

```text
HINDSIGHT_API_RERANKER_PROVIDER=google
HINDSIGHT_API_RERANKER_GOOGLE_PROJECT_ID=<gcp-project-id>
# Optional when not using Application Default Credentials:
# HINDSIGHT_API_RERANKER_GOOGLE_SERVICE_ACCOUNT_KEY=/path/to/service-account.json
```

`deploy/.env.vps.example` can package the local service-account JSON into
the instance secrets, and `deploy/scripts/vps-start.sh` writes it back under
`/home/hindsight/.hindsight/secrets` (same as `HINDSIGHT_HOME` for the `hindsight`
user) before starting Hindsight.

For local smoke testing without an external reranker, Hindsight supports the
`rrf` provider:

```text
HINDSIGHT_API_RERANKER_PROVIDER=rrf
```

In `auto` mode, local scripts start Hindsight only when `HINDSIGHT_API_LLM_*`
configuration is present. Set `JOSHU_HINDSIGHT_ENABLED=true` to require memory
startup and fail fast if Hindsight cannot become healthy. The sandbox stack also sets
`HINDSIGHT_REQUIRE_EXTERNAL_ML=true`, so its slim Hindsight startup requires
`HINDSIGHT_API_EMBEDDINGS_PROVIDER` and `HINDSIGHT_API_RERANKER_PROVIDER`.

The sandbox image runs Hindsight as an unprivileged `hindsight` user and starts PostgreSQL as
the `postgres` user. It persists Hindsight separately from Hermes and ArozOS:

```text
/home/hindsight/.hindsight          # Hindsight profiles/logs/state
/home/hindsight/.cache/huggingface  # local embedding/reranker model cache
/tmp/hindsight-postgres/data        # PostgreSQL data (ephemeral VPS container storage)
```

The sandbox image uses PostgreSQL 15 from Debian packages and builds pgvector from source
during image creation, avoiding pg0's precompiled pgvector/glibc mismatch. The
pgvector Makefile defaults to `-march=native`; Joshu's VPS sandbox image passes
`OPTFLAGS=""` so `vector.so` is portable across image build and runtime CPUs.
Without that, Postgres backends can die with `signal 4: Illegal instruction`
during vector work, which surfaces as Hindsight `hindsight_retain` failures,
async worker errors (`connection was closed in the middle of operation`), and
similar 500s while health and read-only graph endpoints still look fine.

The current PostgreSQL data directory is intentionally ephemeral because earlier
Docker volume attempts rejected PostgreSQL `initdb` permission changes. The sandbox stack also
recycles the web-server container when the function timeout elapses or after idle
scaledown; that wipes `/tmp/hindsight-postgres/data`. Move Hindsight to an
external PostgreSQL service before relying on VPS memory durability.

Only one Postgres process should write to a given Postgres data directory. The
current single-container layout is limited to one container for that reason. Future
multi-sandbox orchestration should allocate separate Hindsight/Postgres volumes
per sandbox, or move to an explicitly shared Postgres service.

To inspect memory behavior, prefer Hindsight APIs, Hindsight Viewer, or the
Hindsight control plane over reading PostgreSQL files directly. Joshu includes a
lightweight **Hindsight Viewer** ArozOS app that calls `/joshu/api/hindsight/*`
and renders simple versions of the constellation graph and entity co-occurrence
graph from the local Hindsight API.

Hindsight Viewer exposes:

- **Constellation**: memory nodes plus typed semantic, entity, temporal, and
  causal links from `/v1/default/banks/{bank_id}/graph`.
- **Entity Co-occurrence**: entity nodes and co-occurrence links from
  `/v1/default/banks/{bank_id}/entities/graph`.
- **Graph mode**: a lightweight SVG node-link view.
- **Table mode**: raw node and edge rows. This is useful because the
  constellation endpoint can return several typed edges for the same source and
  target pair. A small bank can therefore show many raw edges while having only a
  few unique memory pairs.

The repo smoke test:

```bash
npm run hindsight:smoke
```

retains a distinctive fact into `HINDSIGHT_BANK_ID` and recalls it from the local
Hindsight API. A successful recall proves the end-to-end path is writing and
reading through Hindsight's local PostgreSQL storage. Hindsight may normalize or
summarize the retained text, so exact random marker strings are not always
preserved; inspect the returned memories and Hindsight Viewer when debugging.

## Hermes runtime config: local `~/.hermes` vs VPS / image

Three different paths are easy to confuse:

| Path | Role |
| --- | --- |
| **`~/.hermes/`** (your Mac) | Personal Hermes home: full `config.yaml`, sessions, logs |
| **`/root/.hermes/`** (VPS container) | **Persistent Docker volume** `joshu_hermes` — runtime state per customer |
| **`<repo>/.hermes/`** | Repo-only project plugins (`plugins/README.md`); image path `/opt/joshu/.hermes/` |

**Your laptop `config.yaml` is not baked into `joshu-sandbox` images.** Product
behavior is re-applied at boot from repo files, `instance.env`, and Joshu startup
code (`ensureJoshuHermesConfig` in `src/hermesApi.ts`, `ensure_hermes_runtime_config`
in `deploy/scripts/vps-start.sh`).

### Product vs personal config split

Hermes reads a single `$HERMES_HOME/config.yaml`. Joshu splits ownership so box
snapshots and factory reset can keep personal tuning separate from product defaults
([`docs/box-state.md`](box-state.md#hermes-config-split)).

| Layer | File | Keys |
| --- | --- | --- |
| Personal (snapshotted) | `config.user.yaml` | `voice`, `stt`, `tts`, `profiles`, `personalities`, `messaging` |
| Product (Joshu-managed) | in-memory managed slice | `model`, `toolsets`, `skills`, `mcp_servers`, `browser`, `plugins`, `kanban`, … |
| On disk for Hermes | `config.yaml` | Merged product + personal |

Implementation: [`src/hermesConfigSplit.ts`](../src/hermesConfigSplit.ts) +
[`ensureJoshuHermesConfig()`](../src/hermesApi.ts).

**Write policy:** Joshu does **not** rewrite `config.yaml` on every health probe.
`writeMergedHermesConfig()` compares the merged YAML to disk and skips when equal.
When a write is needed, it uses an in-process lock and atomic replace (temp +
rename), matching Hermes upstream `atomic_yaml_write` behavior. Corrupt
`config.yaml` triggers auto-repair: rebuild product defaults, merge
`config.user.yaml`, log `[hermes-api] repaired Hermes config`.

**Do not** hand-edit `config.yaml` on VPS for personal voice/TTS settings — use
Hermes Admin or edit `config.user.yaml` so Joshu can merge cleanly. Product
changes belong in repo code, `instance.env`, or `integrations/hermes/*` — not
only in the volume copy of `config.yaml`.

| Piece | Location |
| --- | --- |
| URL | `https://hermes-admin.${CUSTOMER_DOMAIN}/` (e.g. `https://hermes-admin.patrick.box.joshu.me/`) |

| Boot | `deploy/scripts/vps-start.sh` after Hermes gateway warm-up |

**Env vars** (`/etc/joshu/instance.env`):

```dotenv
JOSHU_HERMES_DASHBOARD_ENABLED=true
JOSHU_HERMES_DASHBOARD_DIRECT=true          # VPS default
HERMES_DASHBOARD_DOMAIN=hermes-admin.${CUSTOMER_DOMAIN}   # optional override
JOSHU_HERMES_DASHBOARD_USER=admin
JOSHU_HERMES_DASHBOARD_PASSWORD=<required on production>
HERMES_DASHBOARD_HOST=127.0.0.1
HERMES_DASHBOARD_PORT=9119
```

**Host header (Caddy → Hermes):** Hermes binds loopback and rejects non-loopback `Host` values ([GHSA-ppp5-vxwm-4cf7](https://github.com/NousResearch/hermes-agent/security/advisories/GHSA-ppp5-vxwm-4cf7)). Caddy must rewrite the upstream header:

```caddy
reverse_proxy 127.0.0.1:9119 {
    header_up Host 127.0.0.1:9119
}
```

Without this, the browser shows `{ "detail": "Invalid Host header. Dashboard requests must use the hostname the server was bound to." }`. See [troubleshooting — Hermes Admin Invalid Host](vps-sandbox/troubleshooting-and-lessons.md#hermes-admin-invalid-host-header).

**Kanban:** served at site root in direct mode — no Joshu path prefix, no kanban WebSocket patch. **`kanban.auto_decompose`** is **enabled** globally. Use skill **`ea-project-kanban`** for **multi-step and HITL** work on `project-<slug>` boards (not mail ingress). **jChat kickoff:** connectors MCP **`project_kanban_ensure_board`** / **`project_kanban_create_triage_root`** (not `scheduling_*`, not CLI when MCP is registered). **Workers:** native **`kanban_*`** (list, block, complete); outbound mail via **`nylas_send_message`**. EA ingress board **`ea-mail-ingress`** uses **assignee → ready** (never triage); legacy **`ea-sched-ingress`** same rule; [`hermes-kanban-bridge.py`](../scripts/hermes-kanban-bridge.py) rejects triage creates on EA ingress boards. **Mail ingest (2026-06-17):** mirror → **`ea-mail-classifier`** → **Triage stub** + **`ea-mail-ingress`** → Patrick files `Projects/<slug>/` + `mail_*` track; optional **`scheduling_*`** child on **`ea-scheduling`**. Default chat toolsets include **`kanban`**, but native `kanban_*` may not appear in every jChat trace — prefer connectors **`project_kanban_*`** for board setup; see [partial MCP catalog](vps-sandbox/troubleshooting-and-lessons.md#partial-mcp-tool-catalog-jchat).

## Updating Hermes

Updating Hermes should not overwrite Joshu customizations when they live in this
repo or in `$HERMES_HOME`.

The safest home for Joshu notes is this repo, not the Hermes checkout:

- Machine-specific runbooks belong in `docs/local-installation.md`.
- Integration policy belongs in this file.
- Repo-owned skills belong in `integrations/hermes/skills/`.
- Skill policy: allowlist in `integrations/hermes/skills-enabled.yaml`; optional explicit blocks in `integrations/hermes/skills-disabled.yaml`.
- Repo-owned plugins belong in `<repo>/.hermes/plugins/`.
- Temporary upstream patches belong in `patches/` and `scripts/hermes-*.patch`
  (Langfuse, skill evolution, content filter, Camofox HITL).

### Joshu update script (recommended)

Use `scripts/update-hermes-agent.sh` to update the **external** Hermes checkout
and keep Joshu's deploy pin in sync. This is the preferred path when promoting a
new upstream **stable release** (GitHub release tag) to local dev and deploy pin.

```bash
npm run hermes:status          # compare local pin/checkout vs latest release
npm run hermes:update          # update + snapshot (or: scripts/update-hermes-agent.sh update)
npm run hermes:rollback        # restore last snapshot
```

What the script does on `update`:

1. Creates a rollback snapshot under `.local/hermes-update-snapshots/` (gitignored).
2. Resolves the latest NousResearch release tag (or `--tag v2026.5.7`) to a commit SHA.
3. Checks out that commit in the external Hermes tree (`HERMES_DIR`, default
   `/Users/danbenyamin/Documents/dev/hermes-agent`) from the `upstream` remote.
4. Reinstalls Hermes editable deps (image-parity extras by default) and restores
   Joshu's pinned Hindsight packages in the Hermes venv.
5. Runs `scripts/apply-hermes-hitl-patch.sh` only when `adopt_existing_tab` is
   missing from `tools/browser_camofox.py`.
6. Re-applies Joshu Hermes patches: Langfuse plugin, skill evolution ledger,
   provider `content_filter` handling (`apply-hermes-content-filter-patch.sh`),
   and agent-loop Langfuse `post_tool_call` (`apply-hermes-invoke-tool-post-hook-patch.sh`).
7. Verifies generic Camofox adoption support (same requirement as `deploy/RELEASE.json`).
8. Updates `HERMES_AGENT_REF` in `deploy/RELEASE.json` and the commented pin in
   `.env.example`.

### Rollback snapshots

Each `update` writes a timestamped directory under
`.local/hermes-update-snapshots/<id>/` and symlinks `latest` to it. Rollback
restores:

| Snapshot file | Restores |
|---------------|----------|
| `deploy/RELEASE.json` | Joshu deploy default `HERMES_AGENT_REF` |
| `.env.example` | Commented `HERMES_AGENT_REF` example |
| `hermes-venv.freeze.txt` | Hermes venv packages via `pip install -r` |
| `manifest.json` | Git commit to `git checkout --detach` in `HERMES_DIR` |

This is enough to undo **Joshu's integration pin** (checkout + venv + deploy pin).
It does **not** snapshot:

- `$HERMES_HOME` (`~/.hermes`) runtime config, auth, or memory data. Use
  `venv/bin/hermes update --backup` in the Hermes checkout when you also need
  Hermes-managed runtime state backed up.
- Uncommitted edits in the Hermes tree (rollback only returns to the saved commit;
  stash or copy local changes before updating if you need them).
- A full `venv/` tarball (`pip freeze` is intentional: smaller and reproducible).

### Manual / Hermes-native updates

When Hermes changes outside the release script:

1. Keep the local Hermes checkout clean; move scratch files out before updating.
2. For runtime state in `$HERMES_HOME`, prefer `venv/bin/hermes update --check`
   followed by `venv/bin/hermes update --backup` over a raw `git pull`.
3. Update the external/local Hermes checkout or `deploy/RELEASE.json` hermesRef.
4. Run Joshu locally.
5. Verify the gateway still starts and discovers Joshu skills/plugins.
6. Verify Hermes still honors `CAMOFOX_USER_ID`, `CAMOFOX_SESSION_KEY`, and
   `CAMOFOX_ADOPT_EXISTING_TAB=true` by operating on the visible Camofox tab.
7. Verify the slim Hindsight local shape still works, then run
   `npm run hindsight:smoke`.
8. If using an older checkout without generic Camofox adoption support, run
   `scripts/apply-hermes-hitl-patch.sh`; it skips cleanly when the generic hook is
   already present.

Prefer upstreaming small generic hooks to Hermes over maintaining a long-lived
Hermes fork.
